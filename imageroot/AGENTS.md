# imageroot Guidelines

- This subtree is the installed NS8 module payload. Keep it aligned to the checked-in per-agent Hermes runtime, which now runs the dashboard and gateway together, plus the shared auth publishing service.
- There is no OpenViking runtime, no hidden backend runtime, and no `AGENTS_LIST` registry anymore. There is now one shared auth publishing target: `hermes-auth.service` plus the generated `authproxy.env`, `authproxy_secrets.env`, and `authproxy_agents.json` files.
- The runtime contract is now: one configured agent equals one metadata file, one generated Hermes env file, one generated Hermes secrets env file under `secrets/` that includes a unique `API_SERVER_KEY`, one per-agent subdir inside the shared `hermes-agents-home` volume mounted at `/opt/agents/<id>/`, one `hermes@<id>.service`, one `hermes-socket@<id>.service`, one `hermes-pod-<id>` pod, one rootless `hermes-<id>` container that runs the dashboard and gateway together, one rootless `hermes-socket-<id>` relay container, and one dashboard socket file at `dashboard-sockets/agent-<id>.sock`. Shared publishing adds one `hermes-auth.service`, one rootless `hermes-auth` container, one shared Traefik route, and the generated shared auth runtime files.
- Keep restart ownership in systemd: `hermes@<id>.service` and `hermes-auth.service` handle restart policy, and the Podman pod/container launches should not add container-level `--restart` policies.
- Long-running containers must be started, stopped, and restarted through `systemctl --user`, not directly from action code except for short one-shot maintenance containers such as home seeding or ownership repair. Preserve `%S/state` working directories, `PODMAN_SYSTEMD_UNIT=%n`, loopback-only published listeners, and `ExecStop` / `ExecStopPost` cleanup in units.
- Preserve named-volume ownership semantics: all agent homes share one `hermes-agents-home` volume with per-agent subdirs at `/opt/agents/<id>/`. Do not regress into per-agent volumes or root-owned agent homes across restarts. Use `ensure-agent-home-ownership` so ownership follows the configured Hermes image's dynamic `hermes` UID/GID instead of a hardcoded numeric UID. `update-module.d/30ensure-agent-home-ownership` owns the best-effort stop, reset, and repair flow, and `update-module.d/80restart` owns restarting the enabled Hermes, socket, and shared auth services afterward.
- Preserve the NS8 action model already used here: numbered executable action steps, JSON stdin for actions, machine-readable JSON stdout for reads, diagnostics on stderr, schema files beside public actions, idempotent configure/destroy/restore flows, and early validation failures that report `validation-failed`.
- `environment` is shared NS8 state. Merge only managed keys and preserve core-managed values such as `HERMES_AGENT_HERMES_IMAGE`, `HERMES_AGENT_AUTH_IMAGE`, and `HERMES_AGENT_SOCKET_IMAGE`.
- Keep secrets out of `environment`, stdout, stderr, and logs. Keep module-wide secrets in `secrets/shared.env`, generated per-agent Hermes secrets in `secrets/<id>.env`, and shared auth runtime outputs in `authproxy.env`, `authproxy_secrets.env`, `authproxy_agents.json`, and `dashboard-sockets/`. This per-agent secrets layout is intentional; do not collapse it into one generic `secrets.env` file.
- Keep shared smarthost discovery in `discover-smarthost`, keep per-agent env and secrets generation in `sync-agent-runtime`, and keep strict first-seed-only home seeding in `configure-module/75seed-agent-home`. `sync-agent-runtime` owns the generated `authproxy_agents.json` `upstream_socket` records. That seed step must use the generated public `agents/<id>/agent.env`, mount checked-in templates read-only, and preserve existing `SOUL.md` plus `.env` content in the agent's subdir inside the shared volume.
- `sync-agent-runtime` now owns the generated per-agent `API_SERVER_KEY`, `AGENT_ALLOWED_USER`, plus `USER_DOMAIN`, `LDAP_HOST`, `LDAP_PORT`, `LDAP_BASE_DN`, `LDAP_SCHEMA`, `LDAP_BIND_DN`, and `LDAP_BIND_PASSWORD` when shared dashboard auth settings are configured.
- Shared dashboard publishing is controlled by `base_virtualhost`, shared `lets_encrypt`, shared `user_domain`, and per-agent `allowed_user`. Routes are owned by `configure-module`, removed by `destroy-module`, and should remain stable under the `<module_id>-hermes-auth` instance name. If you change route payload fields or cleanup behavior, update Traefik-facing actions, tests, UI, and docs together.
- Preserve user-domain behavior through `agent.ldapproxy`, `agent.bind_user_domains`, and LDAP helper filters. If LDAP relation changes should affect running auth services without a configure run, add and document the corresponding NS8 event handler instead of assuming it already exists.
- Do not add provider `srv` Redis keys or `node:fwadm` firewall exposure unless the module becomes a service provider or adds direct public TCP/UDP services. HTTP dashboard access should stay behind Traefik and the loopback auth listener.
- Do not inject the shared `environment` file directly into containers. Per-agent Hermes containers should consume the generated per-agent public and secret env files only, and the shared auth container should consume the generated shared auth env and registry files plus the mounted `dashboard-sockets/` directory only.
- Backup scope is declared in `etc/state-include.conf` (`state/agents`, `state/secrets`, `volumes/hermes-agents-home`). Restore follows the common NS8 pattern: `restore-module/06copyenv` replays only Hermes-managed shared environment keys from `request['environment']`, then `restore-module/20configure` reruns `configure-module` from the restored `agents/` tree. Do not add derived files (e.g. `agents/<id>/agent.env`, `authproxy.*`) to the backup scope. No Redis dump/restore helpers are needed unless this module starts owning Redis keys.
- When adding a new update-module.d migration script, use a number below 30 so it runs before the ownership repair step and before the service restart step.
